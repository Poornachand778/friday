I get what you’re aiming for: a live “context board” that shows the LLM’s current working set—primary scene, supporting snippets retrieved via vector search, camera/calendar cues, etc.—so you can see at a glance what Friday is relying on.

How to build it (incrementally):

Define a Context Registry

Create a small service (e.g. context_manager.py) that maintains a typed list of context items:
{"kind": "scene", "id": "SCN015", "title": "...", "source": "active_edit", "expires_at": ...}
{"kind": "scene", "id": "SCN031", "source": "vector_support", ...}
{"kind": "camera", "payload": "...", "source": "vision_stream"}
Each entry gets an origin (active_edit, vector_support, calendar, camera) and an expiration policy.
Integrate with Existing Flows

When scene_update is called, mark that scene as the active_edit.
Every time scene_search or other retrieve tools are invoked, add/refresh entries with source="vector_support".
Calendar or camera MCP servers would push entries into the same registry when new events arrive.
Expose via MCP Tool

Add a read-only tool like context_get so the agent (or UI) can fetch the current context stack.
Optionally expose context_set_primary so non-scene tools can change focus (e.g., when you explicitly shift to a new scene).
Display to the User

Build a simple dashboard (terminal panel, web UI, or even an overlay in your IDE) that polls context_get and renders:
Primary context (the scene currently being edited).
Supporting context items (retrieved scenes, calendar reminders, camera detections) grouped by source.
Highlight deprecated/expired items so you know background context is being dropped.
Persistence & Sessions

Update the short-term memory JSONL (or Postgres) whenever the registry changes, so the context survives restarts.
Tie entries to a session ID; when you switch projects or restart a conversation, you can reset or load the previous stack.
LLM Prompting

Update the system prompt (or tool instructions) to reference the context board, ensuring the model announces context shifts (“Still working on SCN015; retrieved SCN031 for connective tissue…”).
Optionally require the agent to call context_get before answering, so responses are grounded in the visible context.
This keeps the workflow transparent: you’ll always see the primary scene, plus any extra context the LLM is considering (other scenes, calendar events, camera cues). When you’re ready for voice/camera integrations, those sources just drop new entries into the same registry, and the UI updates automatically.
